# Harness Advent

`Harness Advent` — собственный AI-harness для инженерных задач в репозиториях. Он должен помогать анализировать проекты, отвечать на вопросы по коду, выполнять цепочки разработки и ревью, а также координировать субагентов для реализации фич и приложений.

Проект находится на ранней стадии. Серверный локальный MVP уже реализует безопасный жизненный цикл заданий, SQLite-хранилище, проекты и локальные RAG-источники; реальные внешние интеграции по умолчанию отключены. Возможности ниже задают целевую архитектуру и границы дальнейшего развития.

## Назначение

Harness должен поддерживать следующие сценарии:

- работа с MCP-инструментами и внешними интеграциями;
- RAG по подключённым проектам и их документации;
- ответы на вопросы о кодовой базе с указанием источников;
- управляемые цепочки действий: исследование, планирование, изменение кода, проверки и отчёт;
- код-ревью и работа с pull/merge request: чтение, создание и публикация комментариев через интеграционные адаптеры;
- запуск и координация субагентов;
- использование локальных моделей, прежде всего для RAG, и облачных моделей для задач, где нужна более высокая способность к рассуждению;
- подключение и применение навыков (skills) в повторяемых рабочих процессах.

## Быстрый старт сервера

Требование: JDK 21.

```bash
cd harness-advent-server
./gradlew test
./gradlew run
```

Текущий API, переменные окружения и ограничения адаптеров описаны в [README сервера](harness-advent-server/README.md). При развитии сервера сохраняй провайдер-независимое ядро и включай внешние операции только через явные approvals.

Для работы с одним выбранным проектом в Docker и публикации локального API через ngrok см. [инструкцию запуска](docs/docker.md). Контейнер получает единственный bind mount проекта как `/workspace` и публикует API только на `127.0.0.1:8095`.

## Ассистент проекта

После регистрации и сканирования разрешённого проекта в веб-клиенте доступна команда:

```text
/help Как устроен сервер?
```

Она создаёт фоновую read-only RAG-задачу. В контекст попадают README в корне, `docs/`, API-описания (`openapi`, `swagger`, `api/`, `schemas/`, `contracts/`) и исходные файлы из ограниченного списка расширений; ответ обязан содержать ссылки на путь и строки источника.

MCP подключаются отдельно от `/help` через локальный конфигурационный реестр. Первая запись — официальный GitHub MCP только для чтения и одного явно разрешённого GitHub-репозитория. Настройка и ограничения описаны в [docs/mcp.md](docs/mcp.md).

Для цели вроде «найди все использования API и обнови changelog» создай `agentWorkflow` в режиме `mayModify`. После явного подтверждения изменения файлов ассистент сам ищет и читает несколько файлов, записывает разрешённые текстовые файлы и сохраняет журнал операций с хешами изменённых файлов. Подробнее — в [README сервера](harness-advent-server/README.md).

## Автоматическое ревью PR

Для каждого подключаемого репозитория в него копируется шаблон GitHub Action. Он создаёт read-only задачу `codeReview`: передаёт diff и список изменённых файлов, сервер добавляет RAG-контекст из документации и кода этого проекта и возвращает Markdown-отчёт в Job Summary его PR. Настройка секретов, ограничения и проверка описаны в [docs/code-review.md](docs/code-review.md).

## Безопасность

Секреты, токены, ключи провайдеров, данные подключённых проектов и локальные индексы не должны попадать в Git. Локальная конфигурация хранится вне репозитория или в игнорируемых `.env`-файлах; в репозитории допустимы только шаблоны вида `.env.example` без значений секретов.

Перед коммитом следует проверить `git status`, diff и отсутствие секретов в добавленных файлах. Полные правила приведены в [docs/security.md](docs/security.md).
