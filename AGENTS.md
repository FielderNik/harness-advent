# Инструкции для Codex

## Цель проекта

Harness Advent — платформа для выполнения инженерных AI-задач над подключёнными репозиториями. Серверная часть пишется на Kotlin/Ktor, веб-клиент — на TypeScript/Vite. Документация и пользовательские тексты ведутся на русском языке, если задача не требует другого языка.

Перед архитектурными или интеграционными изменениями прочитай `docs/architecture.md` и `docs/security.md`.

## Границы архитектуры

- Ядро harness не зависит от конкретного поставщика модели, MCP-сервера или платформы pull request.
- Codex — основной агент разработки, но его запуск, другие облачные модели, локальные модели, MCP и Git-провайдеры подключаются через явные адаптеры.
- Локальные модели в первую очередь используются для RAG. Не переноси их ограничения или API в доменную модель.
- Долгие операции запускаются как задания с идентификатором, статусом, журналом и отменой; HTTP-обработчик не должен удерживать их выполнение.
- Субагенты получают минимальный необходимый контекст, ограниченную область работы и проверяемый результат. Их сообщения, команды и изменения должны быть доступны для аудита.
- Доступ к репозиториям, выполнение команд, публикация комментариев и создание pull request — операции с явными разрешениями и границами.

## Правила изменений

- Сначала изучи существующий код, документацию и локальные инструкции.
- Делай минимальные изменения в рамках поставленной задачи; не добавляй абстракции и зависимости без необходимости.
- Не смешивай транспорт Ktor, доменную логику, интеграции, хранение и UI-модель в одном слое.
- Не реализуй веб-функции в серверной части и наоборот. Общие контракты описывай явно.
- Шаблонный код Ktor допустимо заменять только в рамках согласованного этапа; сохраняй Koin, Exposed и kotlinx.serialization, если задача не меняет стек.
- Для каждой законченной возможности добавляй или обновляй русскоязычную документацию: назначение, границы, конфигурацию и способ проверки.

## Секреты и данные

- Никогда не добавляй в Git токены, API-ключи, пароли, приватные URL, содержимое `.env`, локальные базы, RAG-индексы, логи с чувствительными данными или рабочие копии подключённых проектов.
- Не выводи секреты в логи, ответы API, отчёты агентов и тестовые фикстуры.
- Значения конфигурации получай из окружения или секрет-хранилища. В репозитории храни только `.env.example` с пустыми или заведомо безопасными примерами.
- Перед коммитом проверь добавленные файлы и убедись, что игнорируются `.env`, локальные базы, индексы и артефакты сборки.

## Проверка результата

- Выполняй наиболее узкие релевантные проверки: тесты изменённого модуля, компиляцию, линтер или typecheck.
- Для серверного модуля базовая проверка: `cd harness-advent-server && ./gradlew test build`.
- В итоговом сообщении указывай изменённые файлы, что проверено и что пока не запускалось.
