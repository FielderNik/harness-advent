# Генерация unit-тестов для репозиториев

## Назначение и границы MVP

Кнопка «Написать тесты» строит для подключённого проекта план покрытия Kotlin-реализаций репозиториев. В MVP анализируются только файлы по соглашению `src/*Main/**/data/repository/**/*RepositoryImpl.kt`; один запуск выбирает ровно один класс. Это не является общим измерением code coverage и не заменяет статический анализ Kotlin.

В веб-интерфейсе этот сценарий использует профиль `deepseek`, независимо от профиля, выбранного для остальных задач. Перед подтверждением изменения учитывай, что выбранный исходный файл и ограниченный Gradle-контекст будут отправлены во внешний DeepSeek API для подготовки теста.

План хранится в SQLite. Для каждого класса в нём есть исходный и ожидаемый test-путь, статус, причина, задача, ветка и ссылка на PR. Статус `coveredInOpenPr` означает, что подходящий test-файл изменяется в открытом PR; новый дубль для него не создаётся.

## Последовательность

1. Сервер находит реализации репозиториев, существующие тесты и — при включённой GitHub-конфигурации — пути файлов из открытых PR.
2. План создаётся или обновляется. Выбирается первый элемент со статусом `needsTests`; параллельный запуск для того же проекта отклоняется.
3. Пользователь подтверждает `fileModification`. Сервер создаёт изолированный Git worktree в `.harness-worktrees/<task-id>` с уникальной веткой `harness/tests/<class>-<task-id>`. Локальные изменения в основной рабочей копии допустимы: пайплайн их не читает, не меняет и не включает в PR.
4. Модель получает выбранный исходник, контракты его прямых импортов из подключённого проекта и ограниченный Gradle-контекст; этого достаточно для fake DAO и проверки моделей без доступа к произвольным файлам. Она возвращает полное содержимое заранее рассчитанного Kotlin test-файла, не получает shell и не выбирает путь записи.
5. Сервер записывает этот файл, запускает только настроенную Gradle-команду, сохраняет результат и делает локальный commit.
6. Пользователь подтверждает `externalPublication`. Сервер пушит ветку и создаёт GitHub PR, обновляет план и удаляет временный worktree.

При ошибке создания worktree, тестов или GitHub PR не создаётся. Временный worktree удаляется, а элемент плана получает статус `failed`.

## Конфигурация

Значения указываются только в игнорируемом `harness.local.properties` или окружении контейнера:

```properties
testGeneration.enabled=true
testGeneration.allowedRepository=owner/repository
testGeneration.baseBranch=main
testGeneration.checkCommand=./gradlew,:composeApp:testDebugUnitTest,--no-daemon,--no-watch-fs,--max-workers=2
testGeneration.checkTimeoutSeconds=900
# Предпочтительно задавать секрет через HARNESS_GITHUB_TEST_GENERATION_TOKEN.
testGeneration.githubToken=
```

`testGeneration.checkTimeoutSeconds` ограничивает проверку 30–900 секундами (по умолчанию — 900). При превышении лимита Harness останавливает запущенный Gradle-процесс, сохраняет полученный вывод и помечает задачу ошибкой; ветка и PR не создаются. Вывод Gradle публикуется в событиях `testGeneration.checkOutput` по мере выполнения; в артефакте `testCheck` сохраняются последние 50 000 символов, чтобы не потерять финальную диагностику компилятора. При ошибке компиляции или падении теста Harness до двух раз передаёт модели тот же test-файл и диагностический вывод, перезаписывает только этот же файл и повторяет проверку. Таймаут повторно не запускается.

Для Compose Multiplatform используй самую узкую подтверждённую Gradle-цель проекта. Базовая проверка запускает `:composeApp:testDebugUnitTest`: она выполняет `commonTest` на Android, не собирая неподдерживаемые в Linux Apple-цели. Образ содержит Android SDK API 36 и build-tools 35.0.0 и 36.0.0; запускается не более двух Gradle workers. Проекты с другой версией `compileSdk` требуют соответствующего расширения образа. В частности, при ошибке Kotlin о nullable receiver модель должна сохранить nullable-значение в локальную переменную через `assertNotNull` перед сравнением; это важно для полей типа `ByteArray?`.

Токен GitHub должен быть fine-grained, ограниченным одним `owner/repository`, с минимальными правами на чтение pull request, запись содержимого и создание pull request. Он никогда не отправляется в браузер, модель, артефакты или логи.

После отдельного подтверждения `externalPublication` сервер создаёт PR через GitHub REST API. Запрос ограничен `testGeneration.allowedRepository`, веткой формата `harness/tests/*` и `testGeneration.baseBranch`; он использует тот же fine-grained токен, который нужен для `git push`. GitHub MCP может оставаться read-only и не требует Docker внутри контейнера Harness.

## Проверка

1. Запусти Harness в Docker, смонтировав только Trainingdiary как `/workspace`.
2. Зарегистрируй `/workspace`, нажми «Написать тесты» и проверь таблицу плана.
3. Подтверди изменение, дождись Gradle-проверки, затем отдельно подтверди публикацию.
4. Проверь созданный PR и запись `prCreated` в плане.

Для Harness: `cd harness-advent-server && ./gradlew test`; для веб-клиента: `cd harness-advent-web && npm run typecheck && npm test`.
