# План реализации сервера

## Назначение и границы

`harness-advent-server` — единственная доверенная точка для выполнения задач, доступа к подключённым репозиториям и интеграций с агентами, моделями, MCP и Git-провайдерами. HTTP-обработчики создают задания и возвращают их состояние; они не удерживают длительное выполнение.

В первую версию не входят многопользовательский SaaS, распределённая очередь и публикация в Git-провайдеры без явного подтверждения. Сначала поддерживается локальный однопользовательский запуск с проектами, разрешёнными в конфигурации.

Ядро не зависит от Codex, облачного поставщика или протокола конкретного MCP-сервера. Они подключаются через порты и адаптеры.

## Целевой поток

1. API валидирует запрос, проект и политику доступа.
2. Сервис создаёт задание и сохраняет его начальное состояние.
3. Фоновый исполнитель берёт задание, создаёт шаги и записывает события и артефакты.
4. Адаптеры вызываются только через разрешённые порты.
5. Клиент читает состояние, журнал и артефакты через API и SSE.

## Модульная структура

После замены Ktor-заготовки код следует разнести по пакетам:

- `api` — Ktor routes, DTO, преобразование ошибок, авторизация и SSE;
- `domain` — задания, сценарии, разрешения, политика передачи контекста и инварианты;
- `application` — use cases: создать/отменить задание, выполнить шаг, подтвердить действие, получить журнал;
- `persistence` — реализации репозиториев на Exposed и миграции;
- `adapters` — Codex, локальные и облачные модели, MCP, Git и файловая система проектов;
- `bootstrap` — Koin-модули, конфигурация и запуск фоновых исполнителей.

Ktor DTO не должны использоваться внутри `domain`, а SDK и HTTP-клиенты провайдеров не должны попадать в `application`.

## Основные данные и контракты

Минимальные сущности:

- `Project`: разрешённая локальная рабочая копия, правила и статус сканирования;
- `Task`: сценарий, проект, режим (`readOnly` или `mayModify`), состояние и автор задачи;
- `TaskStep`: стадия, исполнитель, входной контекст, результат и состояние;
- `TaskEvent`: хронология с временем, уровнем, типом и безопасным payload;
- `Artifact`: отчёт, diff, результат проверки, цитата RAG или ссылка на файл;
- `Approval`: явное решение пользователя для передачи контекста, выполнения команды, изменения файлов или публикации;
- `ModelProfile`: доступный провайдер, модель, лимиты, возможности и политика контекста.

Первая версия API размещается под `/api/v1`:

- `GET /health`;
- `GET, POST /projects` и `GET /projects/{id}`;
- `GET, POST /tasks`, `GET /tasks/{id}`, `POST /tasks/{id}/cancel`;
- `GET /tasks/{id}/events` — SSE;
- `GET /tasks/{id}/artifacts`;
- `POST /tasks/{id}/approvals`;
- `GET /model-profiles` и `GET /model-profiles/{id}/models`.

В ответах API не передаются секреты, заголовки авторизации и неочищенные журналы сторонних клиентов.

## Этапы

### 0. Базовый каркас и конфигурация

- Заменить демо-маршруты, `UsersService` и жёстко заданный H2 URL на каркас модулей.
- Описать `ApiError`, request id, валидацию и `/health`.
- Добавить безопасный `.env.example`, конфигурацию через environment и Koin-модули.
- Завести миграции и интерфейсы репозиториев; выбор локальной БД изолировать реализацией репозитория.
- Добавить тесты маршрутов и use cases.

Критерий: сервер запускается без секретов, `/health` доступен, тесты не используют сетевые интеграции.

### 1. Жизненный цикл заданий

- Реализовать состояния `queued`, `running`, `waitingApproval`, `completed`, `failed`, `cancelled`.
- Создать хранилище задач, шагов, событий и артефактов.
- Запускать работу в сервисе исполнителей с coroutine scope, а не в Ktor route.
- Реализовать отмену, идемпотентность запросов и SSE для обновлений.

Критерий: тестовое задание создаётся, генерирует события, завершается или отменяется без блокировки HTTP-запроса.

### 2. Проекты и локальный RAG

- Зарегистрировать только явно разрешённые каталоги.
- Добавить безопасное перечисление файлов, исключения для `.git`, секретов, build-артефактов и бинарных данных.
- Собирать документы и исходники с метаданными: путь, revision, диапазон строк и хеш.
- Создать локальный индекс, поиск и политику удаления индекса.
- Возвращать RAG-источники отдельными артефактами; ответ агента не подменяет ими доказательства.

Критерий: вопрос по разрешённому проекту возвращает ответ вместе с точными источниками файлов.

### 3. Агенты и skills

- Определить нейтральный `CodeAgentRunner`: запуск, события, отмена, артефакты и usage.
- Добавить адаптер Codex как первый рабочий исполнитель.
- Ввести `SkillRegistry` и проверяемый снимок передаваемых агенту инструкций.
- Реализовать последовательность `research → plan → execute → validate → report` и раздельные права шагов.

Критерий: один локальный сценарий проходит все стадии, а его команды, изменения и проверки видны в аудите.

### 4. Локальные и облачные модели

Первым общим адаптером будет `OpenAiCompatibleModelProvider`: он покрывает локальный Ollama и облачные OpenAI-compatible API. Для поставщиков с отличающимся протоколом добавляется отдельный адаптер, а не условия в ядре.

Порт модели должен поддерживать:

- получение каталога моделей и capability-маркеров;
- запуск запроса, поток событий и отмену;
- нормализованные usage, лимиты, стоимость и ошибки;
- healthcheck без раскрытия ключа.

Сначала подключаются `Local`, DeepSeek и OpenRouter как независимые конфигурационные профили. Имена моделей и цены не фиксируются в доменной модели: каталог и pricing кэшируются адаптером с временем актуальности.

Перед облачным вызовом сервер формирует manifest контекста: выбранные источники, объём, назначение и применённые правила фильтрации. Политики:

- `localOnly` — выход во внешнюю сеть запрещён;
- `metadataOnly` — разрешены только метаданные задания;
- `selectedSources` — передаются только подтверждённые пользователем фрагменты;
- `approvedTaskContext` — передаётся подготовленный и подтверждённый контекст задачи.

Сервер отбрасывает секреты, `.env`, приватные ключи, Git-служебные данные и запрещённые пути до подготовки payload. В аудит сохраняются provider, модель, лимит, usage, стоимость, результат фильтрации и хеш payload; сам ключ и содержимое чувствительного контекста не журналируются.

Критерий: модель можно выбрать и выполнить через сервер, а тесты доказывают отсутствие ключей в API, логах и событиях.

### 5. MCP, review и Git

- Подключить MCP только из контролируемого реестра с ограничением методов и прав.
- Добавить read-only code review: файлы, diff, замечания, ссылки на доказательства и результаты проверок.
- Добавить Git- и pull/merge-request-адаптеры после read-only сценария.
- Любая публикация комментариев, ветки или PR требует отдельного `Approval`.

Критерий: review не изменяет репозиторий; внешний побочный эффект невозможен без подтверждения.

## Конфигурация и безопасность

- Секреты хранятся только в окружении сервера или внешнем секрет-хранилище.
- UI получает только признак готовности профиля, доступные модели и безопасную диагностику.
- Для каждого провайдера задаются endpoint, reference на секрет, лимиты, таймауты и политика передачи контекста.
- Логи проходят редактирование до сохранения; ошибки провайдера не возвращают заголовки и исходные запросы.
- Все изменяющие операции требуют политики или подтверждения пользователя.

## Проверки каждого этапа

Минимальная серверная проверка:

```bash
cd harness-advent-server
./gradlew test build
```

Дополнительно для новых маршрутов нужны Ktor integration tests, для use cases — unit tests с fake-адаптерами, а для облачных адаптеров — contract tests на mock HTTP server. Реальные ключи и реальные облачные вызовы не являются частью автоматической сборки.
