# Безопасность и работа с секретами

## Что считается секретом

К секретам относятся API-ключи, токены облачных моделей, MCP-токены, пароли, SSH-ключи, сертификаты, cookie, приватные URL, персональные данные, содержимое закрытых репозиториев и любые данные доступа к Git-провайдеру.

## Обязательные правила

- Не добавлять секреты в исходный код, тестовые данные, документацию, скриншоты, логи, сообщения агентов и коммиты.
- Хранить локальные значения в окружении, локальном секрет-хранилище или игнорируемом `.env`-файле. Коммитить можно только `.env.example` без реальных значений.
- Не передавать облачной модели содержимое проекта, секреты или персональные данные без явной политики и разрешения пользователя.
- Не включать локальные базы, RAG-индексы, кэш моделей, журналы заданий и рабочие копии подключённых проектов в Git-репозиторий.
- Маскировать значения секретов в любых диагностических сообщениях.
- Выдавать интеграциям минимально необходимые права и по возможности использовать токены с ограниченным сроком действия.
- При контейнерном запуске монтировать в Harness только один выбранный проект, не передавать Docker socket, домашний каталог или SSH-ключи. Значение разрешённого пути должно задаваться окружением контейнера, а не редактируемым файлом настроек.
- Сценарий генерации тестов создаёт Git worktree только внутри смонтированного проекта, требует чистую основную рабочую копию и удаляет временный worktree после завершения. GitHub-токен для публикации передаётся только через секрет окружения, ограничивается одним репозиторием и не попадает в модель, API, артефакты или логи.

## До коммита и публикации

1. Проверь `git status` и diff добавляемых файлов.
2. Убедись, что `.env`, ключи, базы, индексы, журналы и артефакты сборки игнорируются.
3. Проверь, что примеры конфигурации не содержат рабочих токенов или приватных адресов.
4. Не публикуй результат агента, комментарий к pull request или лог, пока он не проверен на чувствительные данные.

## Реакция на утечку

Если секрет попал в файл, лог, историю Git или внешний сервис, его нужно считать скомпрометированным: отозвать или заменить, ограничить доступ к данным, удалить секрет из актуального состояния и зафиксировать инцидент. Простого удаления строки из последующего коммита недостаточно.
