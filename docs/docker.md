# Запуск одного проекта в Docker

Этот режим запускает один экземпляр Harness Advent для одной рабочей копии. Сервер видит выбранный проект только как `/workspace`; другие каталоги хоста в контейнер не монтируются.

## Подготовка

1. Скопируй `harness-advent-server/harness.local.properties.example` в `harness-advent-server/harness.local.properties` и настрой модель. Файл не добавляй в Git.
2. Отключи `mcp.servers` в этом файле, если MCP требует Docker: Docker socket намеренно не передаётся контейнеру.
3. Скопируй `.env.example` в локальный `.env` рядом с `docker-compose.yml`:

   ```properties
   HARNESS_PROJECT_PATH=/абсолютный/путь/к/одному/проекту
   ```

   Docker Compose монтирует именно этот каталог в `/workspace` с правом записи. Не указывай общий каталог, содержащий несколько проектов.

Если локальная модель запущена на хосте macOS/Windows, в `harness.local.properties` укажи для неё `models.local.endpoint=http://host.docker.internal:11434/v1`, а не `127.0.0.1`: внутри контейнера этот адрес указывает на сам контейнер.

## Запуск и ngrok

```bash
docker compose up --build
ngrok http 8095
```

Compose публикует `8095` только на `127.0.0.1`, поэтому ngrok на той же машине может создать туннель, а сервер не открыт напрямую в локальной сети. Используй выданный ngrok URL для веб-клиента или API.

В интерфейсе регистрируй проект с путём `/workspace`. Переменная `HARNESS_ALLOWED_PROJECTS=/workspace` задаётся самим Compose и имеет приоритет над `harness.local.properties`, поэтому через API нельзя зарегистрировать другой путь. Для файлового сценария создай `agentWorkflow` в режиме `mayModify` и подтверди `fileModification`: агент использует только операции поиска, чтения и записи относительно `/workspace`, а журнал и хеши изменений появятся в артефактах задачи.

## Граница доступа

- образ запускается непривилегированным пользователем, с read-only root filesystem, без Linux capabilities и без Docker socket;
- изменяемы только `/workspace`, временный `/tmp` и отдельный volume с SQLite в `/var/lib/harness`; `/tmp` помечен `exec`, поскольку SQLite JDBC загружает из него нативную библиотеку;
- внутренние файловые инструменты принимают лишь относительные пути внутри зарегистрированного проекта; проверка путей в приложении остаётся дополнительной защитой;
- контейнер не получает SSH-ключи, домашний каталог, `.env` выбранного проекта или другие каталоги хоста.

Docker-изоляция защищает от доступа к другим каталогам хоста, но не заменяет авторизацию публичного API. Ngrok URL следует считать внешней точкой доступа: для реального использования добавь аутентификацию на уровне ngrok или reverse proxy.

## Остановка

```bash
docker compose down
```

Состояние Harness (SQLite) остаётся в именованном volume `harness-state`. Изменения файлов сохраняются непосредственно в выбранной рабочей копии через bind mount.
