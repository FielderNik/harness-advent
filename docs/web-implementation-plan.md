# План реализации веб-клиента

## Назначение и границы

`harness-advent-web` — интерфейс для создания и наблюдения за инженерными задачами, разрешениями, результатами поиска, ревью и журналом выполнения. Клиент не запускает команды, не имеет доступа к файловой системе подключённых проектов и не хранит ключи облачных моделей.

Базовый стек — TypeScript и Vite. Для экранов с долгоживущим состоянием задач и потоковыми событиями рекомендуется React; если будет выбран vanilla TypeScript, API-контракты и состав экранов не меняются.

Визуальные правила и адаптация дизайна из `week_6_task_26` описаны в [документации по дизайну](web-design.md).

## Принципы UI

- Сервер — единственный источник истинного состояния задачи; локальное состояние клиента не подменяет его.
- В UI ясно разделены `read-only`, изменение рабочей копии и внешняя публикация.
- Перед побочным действием пользователь видит область действия, данные и получает возможность отмены.
- Потоковые события отображаются как журнал с уровнем, стадией, временем и источником.
- API-ключи не показываются, не вводятся и не попадают в browser storage.

## Структура модуля

- `src/api` — typed HTTP-клиент, DTO, SSE и обработка ошибок;
- `src/features/tasks` — список, создание, детали, timeline и отмена;
- `src/features/projects` — подключённые проекты, статус индекса и источники RAG;
- `src/features/approvals` — просмотр и подтверждение действий;
- `src/features/models` — доступные профили, модели, лимиты и политика контекста;
- `src/features/review` — diff, замечания и проверки;
- `src/components` — только переиспользуемые элементы представления;
- `src/pages` — композиция маршрутов и экранов.

## Основные экраны

### Задачи

- список задач с фильтрами по проекту, сценарию, статусу и времени;
- мастер создания: проект, сценарий, режим доступа, модельный профиль и первоначальное описание;
- карточка задачи: текущее состояние, активный шаг, live timeline, отмена и итог;
- вкладки артефактов: план, источники RAG, команды, diff, проверки, отчёт и ошибки.

### Проекты и RAG

- список разрешённых проектов и статус сканирования;
- просмотр результатов поиска с путём, диапазоном строк и ссылкой на источник;
- объяснение, какие источники были включены в контекст задачи.

### Разрешения и облачные модели

- очередь ожидающих подтверждений;
- диалог передачи контекста в облако: провайдер, модель, список источников, объём, политика и лимит стоимости;
- каталог моделей выбранного server-side профиля с доступностью, возможностями, контекстным окном и ценой;
- отображение фактического usage и стоимости после запуска.

Ключи поставщиков на этом экране отсутствуют. UI может показать «профиль не готов», но не причину, раскрывающую секрет или внутренние настройки.

### Review и публикация

- diff и замечания с привязкой к файлу и строке;
- статус проверок и артефакты агентов;
- отдельный диалог подтверждения для комментариев, ветки или pull/merge request.

## Взаимодействие с API

Клиент использует `/api/v1` как единственную границу. Для долгих задач:

1. `POST /tasks` сразу возвращает `taskId` и начальное состояние.
2. Экран деталей подписывается на `GET /tasks/{id}/events` через SSE.
3. После переподключения клиент повторно запрашивает задачу и артефакты, затем продолжает поток.
4. Если SSE недоступен, допустим ограниченный polling `GET /tasks/{id}` с явным статусом в UI.

`POST /tasks/{id}/cancel` и `POST /tasks/{id}/approvals` всегда показывают промежуточное состояние до серверного подтверждения. Ошибки API отображаются по безопасному `code` и пользовательскому сообщению; технический request id доступен для диагностики.

## Этапы

### 0. Каркас

- Создать Vite-модуль, базовую маршрутизацию, layout, typed API-client и конфигурацию `VITE_API_BASE_URL` без секретов.
- Добавить экран health/status и единый вид ошибок/загрузки.
- Настроить typecheck, unit tests и production build.

Критерий: клиент собирается и безопасно отображает недоступность сервера.

### 1. Задачи и поток событий

- Реализовать список, создание и экран деталей задачи.
- Подключить SSE, timeline, отмену и отображение финального отчёта.
- Добавить пустые состояния, reconnect и polling fallback.

Критерий: пользователь видит изменения состояния задания без ручного обновления страницы.

### 2. Проекты, источники и RAG

- Реализовать выбор проекта в мастере задачи.
- Показать источники и фрагменты с точным путём и диапазоном строк.
- Разделить в интерфейсе цитаты источников и выводы агента.

Критерий: результат поиска можно проверить, открыв указанный источник.

### 3. Агенты, skills и артефакты

- Добавить выбор разрешённого сценария и skills.
- Отобразить стадии, команды, изменения файлов, diff и результаты проверок.
- Явно маркировать режимы read-only и may-modify.

Критерий: пользователь понимает, какой шаг выполняется и что именно изменил агент.

### 4. Локальные и облачные модели

- Получать профили и каталог моделей только от сервера.
- Реализовать выбор профиля, модели и лимитов, если это разрешено серверной политикой.
- Перед отправкой облачного контекста отображать manifest и запрашивать подтверждение.
- Показать streaming status, usage, стоимость, отмену и нормализованную ошибку.

Этот этап переносит полезные функции `week_6_task_26` — выбор DeepSeek/OpenRouter/Local, каталог моделей, usage и стоимость — но исключает browser-side вызовы к поставщикам и Vite-переменные с ключами.

Критерий: UI не может получить API key даже при просмотре собранного приложения или browser storage.

### 5. Review, MCP и публикация

- Добавить экраны замечаний, diff и результатов MCP-инструментов.
- Реализовать подтверждение внешней публикации и отображение итогового статуса.

Критерий: внешнее действие не выполняется при закрытии диалога или отсутствии подтверждения.

## Состояния и доступность

Для каждой загрузки должны быть `loading`, `empty`, `error`, `ready` и, где применимо, `reconnecting`. Отмена и подтверждение должны быть доступны с клавиатуры, а критические статусы — объявляться через доступные live-регионы. Визуальные статусы не являются единственным источником информации: текст статуса и время события выводятся явно.

## Проверки

На старте модуля необходимо добавить команды:

```bash
cd harness-advent-web
npm run test
npm run typecheck
npm run build
```

Минимальный набор тестов: API-client, преобразование состояний задачи, SSE reconnect/fallback, диалог approval и отсутствие секретов в сохраняемых настройках. Для ключевых путей нужны browser tests: создать задачу, увидеть событие, отменить её и подтвердить/отклонить передачу контекста в облако.
